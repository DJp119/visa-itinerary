# CEO Plan: Visa Itinerary — Building a visalist.io clone with LLM-displacement-proof moats

Branch: main | Mode: SELECTIVE EXPANSION

## Vision

### Strategic Thesis
> **Visalist.io answers "what are the rules?" 
The clone answers "what do I do, and what gets me to a stamped passport?"**
>
> The site becomes a dependency because it (a) remembers you, (b) does the paperwork, and (c) shows you what actually worked for people like you last week.

The LLM is the on-ramp. The site is the destination. The bridge is "you can act on this advice without leaving."

### Platonic Ideal
A user with a new visa need opens the site, logs in once, fills a profile (10 fields, 90 seconds), and then:
- Sees a personalized dashboard of every visa they can apply for *right now*
- Uploads passport + bank statement + ITR once, the system extracts and verifies the data
- Asks the AI chat about a specific country, gets an answer that references their actual profile
- One click later, the AI drafts a cover letter pre-filled with their data, downloadable as PDF
- One click later, an application tracker is created with their appointment, a checklist of what's missing, and a reminder schedule
- Two months later, they come back to apply for a different country, and the site already knows everything about them

That user will not switch to ChatGPT. ChatGPT will not write the cover letter with their bank balance. ChatGPT will not remember the Schengen approval from last quarter when they're applying for a UK visa.

### 10x Check
A 10x version of this product is the **travel-document operating system**:
- Profile + vault + generators + tracker are not features, they're a **persistent user state** that compounds
- Every interaction adds to the user's "visa readiness graph" — passport expiry alerts, upcoming renewals, family member profiles, travel history that strengthens future applications
- The site becomes the place where the *next* visa application starts, because the data is already there
- The community signals (Phase 5) create a network effect that no LLM can replicate without a comparable data collection

## Accepted Scope (added to this plan)

The v1 surface, in buildable order:

**Phase 0 — Foundation**
- Astro 6 + Tailwind + design tokens
- Convex backend + auth (email + Google)
- Country directory: 80-120 countries, core visa rules
- SEO routes: `/visa/{country}`, `/passport/{country}` with structured data

**Phase 1 — Profile (the moat)**
- Persistent traveler profile: nationality, passports, prior visas, travel history, family, employment, income
- "Personalized for you" eligibility view
- Logged-in homepage = the personalized view

**Phase 2 — Vault + Generators (the work)**
- Document vault: upload, OCR, verify, reuse
- Generator studio: cover letter (Schengen, US, UK, Canada), NOC, invitation letter, day-by-day itinerary, flight/hotel reservation, financial summary
- Each generator pre-fills from the vault

**Phase 3 — AI Chat (the bridge)**
- Per-country AI chat, behind auth
- System prompt includes user profile
- AI advice is actionable: "add to vault" / "generate this document" buttons inline

**Phase 4 — Tracker (the state)**
- Application state machine: `preparing -> submitted -> in_review -> decision`
- Per-state checklist
- Email reminders
- Timeline that AI and human can both post to

**Phase 5 — Trust signals (deferred, multi-month play)**
- Recent-applicant reports
- Aggregated outcomes ("89% of Indian Schengen applicants in March approved")
- Embassy playbooks
- Real-time appointment slot data (if feasible)

**Phase 6 — Polish (deferred)**
- Family profiles, public travel history, document translation, expiry reminders, mobile polish

### Why Astro + Convex
- Astro's islands keep the public SEO surface cheap (countries, visa rules, blog) and only hydrate the dashboard/chat/builder where the user actually interacts.
- Convex gives you a real reactive backend (profiles, document state, application tracking) without standing up Postgres + auth + storage + realtime yourself.
- If you prefer Supabase, the same shape works — Convex is just less ceremony for the realtime + functions combination.

## Constraints Acknowledged

- Backend with user accounts → Convex (or Supabase) — committed
- Real-time visa data → start with curated seed, add scrapers later
- Crowdsourced data → out of scope for v1, Phase 5
- PDF generation library + file storage → pdf-lib + Cloudflare R2 (or S3)

## Risks

- **Cold start on community signals** (Phase 5) — needs a critical mass of contributors. Mitigation: seed with curated embassy playbooks and use official sources where possible.
- **Data accuracy / liability** — wrong visa info can ruin a trip. Mitigation: "last verified" timestamps, source links, "this is not legal advice" disclaimers, versioned rules.
- **PDF format brittleness** — embassy-specific formats vary. Mitigation: templates per embassy, user-editable, versioned.
- **OCR quality** — passport scans vary wildly. Mitigation: manual review step before data is used in a generator.
- **LLM hallucination in document drafts** — particularly risky for legal/financial documents. Mitigation: pre-filled from verified vault data, user must explicitly edit and approve before generation.

## Success Metrics (post-launch)

- **Dependency signal**: % of users who return within 30 days
- **Profile completion rate**: % of signups who finish the 10-field profile
- **Generator usage**: cover letters generated per user per quarter (target: >= 2)
- **Application tracker adoption**: % of generators followed by a tracker entry
- **AI chat -> action conversion**: % of chat messages that lead to a vault add, generator run, or tracker update

## What I'd Build First (if forcing me to pick one)

Phase 1 (the profile) + the personalized homepage. Two reasons:

1. It's the smallest slice that creates the moat. The personalized eligibility view is something an LLM cannot replicate in a single chat — it requires the user's full travel history and visa record.
2. It's the dependency injection. Once a user has filled in their profile, the cost of switching away is high enough that they'll come back for the next application.

Phases 2-4 are obvious winners but Phase 1 is the foundation they sit on. Build the dependency, then build the work on top of it.

## NOT in scope

- Visa application submission itself (the site advises + generates, doesn't submit to embassies)
- Legal advice / official consular services
- Travel insurance, flight booking, hotel booking (just generate the reservation documents for visa applications, partner with providers if scaling)
- Multi-language UI (start English, add Hindi / Mandarin later if metrics support it)
